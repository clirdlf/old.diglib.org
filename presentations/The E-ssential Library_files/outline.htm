Leading edge services need to be:

  • Highly responsive to multiple audiences
  • Nimble and malleable
  • Available on multiple devices as delivery needs mature
  • Able to interact richly across vendors in a data aggregation service (such as a library)
  • Able to harness the essential skills of the librarian
3
Leading edge services should not be:
  • Monolithic
  • Proprietary
  • Isolated from other collections  data that plays badly with others
  • Built without vendor/library dialogue
  • Built without library/user dialogue
4
The New Realities
  • It is not a web world only any more  users are beginning to expect ebook, PDA, print-on-demand, wireless.
  • Content needs to conform to user needs, and not vice versa.
  • Libraries are losing users to Google, Amazon, and Blackboard  we need to reclaim them
  • If we build it we will come is too often true of what we produce  web services for the ideal user (us)
5
What do we need?
  • Persistence of data -- archive-ready
  • Aggregation of data -- no more isolated silos
  • Partnerships -- publishers need to provide better tools to help libraries market the data to the library user
6
Solutions
  • Standards
  • Partnerships with data providers
  • Partnerships with data users
  • More Standards
7
What is XML and why does it work?
  • It is a core standardized document format
  • It is nimble  it makes re-shaping content easier
  • It  is the hub from which many spokes radiate -- HTML, PDA, ebook, and PDF versions can all be derived inexpensively from the base xml file
  • You do not need to predict all the delivery formats you require when you build the core hub document
8
Conclusions
  • Leading-edge services are not simply delivered from a vendor to a customer  the library is a necessary intermediary
  • Leading-edge services need to be tailored and modified close to the point of delivery
  • Content in leading-edge services needs to be leading-edge in its format
  • XML and standardized metadata proves itself daily
