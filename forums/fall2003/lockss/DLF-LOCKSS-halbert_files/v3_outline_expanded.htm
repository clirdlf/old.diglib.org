Historical Approach to Print Journals Preservation
Libraries independently subscribe to journals, retaining back-issues
Libraries owned the specific content purchased
Interlibrary loan for issues that the library didn’t own, or for particular back-issue content that becomes damaged or lost

The Problem with
E-Journals
Content in electronic journals is effectively leased, rather than owned, by libraries
No back issues archived in the stacks
Because libraries do not own the content, standard approaches to preserving access is disrupted
Potential exists for libraries to lose           e-subscriptions and be left with nothing

Enter the Tortoise
LOCKSS is a collaborative project led by Stanford University libraries to develop tools and consortia models to address the long term retention problems of e-journals;
Led by Vicky Reich; support from Mellon, NSF, Sun, Stanford Libraries
Emory and Indiana libraries have partnered with Stanford to work on specific sub-problems

LOCKSS Mission 1
Build tools and provide support to:
Libraries, so they can easily and affordably build, preserve, and archive local                   e-collections
Own rather than lease electronic information
Retain traditional custodial role of scholarly information

LOCKSS Mission 2
Build tools and provide support to:
Publishers, so they can easily and affordably provide content for preservation and archiving
With minimal risk to their business model or to their publishing platforms
Relinquish responsibility to provide perpetual access
Fulfill librarians’ requirements to guarantee long-term access

LOCKSS Strategy
Create an affordable system whereby libraries can act in a decentralized way to cooperatively maintain local copies of      e-journal content
Lots of copies keeps stuff safe, because the loss of any particular content is recoverable from other sites that own the content

LOCKSS System
Simple to operate & affordable local caching system for e-journal web content
If access to e-journal content is disrupted, the local cache is available for seamless continuity of access
If content in the local cache is disrupted, the system automatically rebuilds the cache from other validated caches

LOCKSS & Proxy Servers
LOCKSS can act alone, or be integrated with library proxy servers
Emory portion of LOCKSS project is to perform test integrations and document results
So far have successfully integrated LOCKSS with both EZProxy and Squid (also PAC files)

"Pros"
Pros
Will allow relatively sophisticated conditional handling of requests
Can cache content to minimize network bandwidth load
Open Source Software, strong developer community
Downsides:
Relatively complex to configure

Squid Example

"Pros"
Pros
Simple to configure
LOCKSS works with EZ Proxy 2.2
Will allow you to forward requests to the LOCKSS server, which then mediates the request
Downsides:
No conditional handling of requests
EZ Proxy does no caching for bandwidth purposes
Relatively simple authentication configuration

Proxy Auto Configuration (PAC) Files
PAC files set up on public access workstations with web browsers
LOCKSS works fine for PAC files
WPAD (Web Proxy Auto Detection) will eventually supercede PAC files; don’t know if LOCKSS will work with it or not (haven’t tried)
