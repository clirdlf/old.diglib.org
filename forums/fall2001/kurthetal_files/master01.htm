Our presentation draws on the Cornell University Library’s work with the ENCompass digital library management system, which at this point is a development rather than an implementation effort.  In that light, Sarah and I will raise more questions than we will answer.  Particularly because we are still in development, we welcome your questions and observations and we would enjoy talking with you about our work.
1.This afternoon I’ll begin by taking a look at data structures within ENCompass in order to better understand how they impact the presentation of granularity in object displays.
2.Then Sarah will explore ways in which searching and browsing displays can present relative granularity and structural relationships in order to increase users’ ability to know what they’ve found and determine whether it will meet their needs.
The digital collection environment we’re seeking to bring to our users is multiply heterogeneous.  Its subject content has the potential to span all areas in which the library collects.  Central to our discussion today, the granularity of the objects targeted by system records ranges from one page of a digitized pamphlet to the search interface for Early English Books Online, which in turn opens onto over 100,000 digitized books and serials.  The native metadata sets we’re importing vary as well.  So far we’ve been working with MARC, TEILite, EAD, DC, and locally developed element sets. By saying the environment is heterogeneous in terms of penetration, I mean that we are variously able to provide direct access to the objects described by the metadata we’re importing.  In the case of MARC records describing physical artifacts in the CUL collection, the penetration begins and ends with the metadata itself.  EAD finding aids for undigitized collections go farther in representing content, as do records linked to scanned images of three-dimensional objects, for example.  And, records capable of delivering born-digital texts or images go farther still. Finally, the format of represented objects varies to the extent offered by past and present technologies. An important outcome of looking at the collection environment in this way is to recognize that each of these facets intersects with the others to a greater or lesser extent.  Depth of penetration as we’ve seen is closely linked to resource description and format.  Similarly, we can’t address aspects of granularity without considering how they interact with other environmental factors.
Working with ENCompass to bring collections from this complex environment to our users, four system elements emerge as important.  Sarah and I will explore them in some detail. First, the repository structure into which we import resource descriptions and their associated resource links lives separately in the system from the collection structure which organizes resources for presentation to users. Second, because object record displays are controlled by XSL, we have a relatively high degree of control over them. Third, the collection structure is hierarchical, which offers a means for establishing relationships among digital objects. Finally, in addition to serving as a path to lead users to digital objects, the system’s searching-and-browsing interface has the capacity to serve as a communications tool to tell users about digital objects and the relationships among them.  We expect that this capacity of the user interface will be significantly enhanced in the next version of the ENCompass software, Version 2.0, when the search-and-browse interface is driven by XSL as object record displays are now.
Using ENCompass to bring digital content to users is providing those of us working in the development effort with two significant opportunities:
It’s giving us a chance to start with a blank slate as we arrange digital objects within the collection structure, to develop a display language of words and icons to RE-present content in meaningful ways, and then to deliver that content to the screens of our users.
And, by virtue of our role as a development partner, we’re able to help shape the evolution of ENCompass based on our own use of the system and eventually based on what we will be able to learn from the ways in which end users interact with the system.
With those opportunities in mind, let’s look at aspects of our work with ENCompass so far that bear on how we can address mixed granularity across collections. This is the ENCompass Relationship Manager window, which uses two side-by-side expanding outlines, with repositories on the right and collections on the left.  As I mentioned, the separation of repositories from collections is important to our discussion today. On the repository side of the system, we can use repository-creation and metadata-loading scripts to import metadata records from a common file source with a common element set and store them as containers and objects within a repository.  The metadata element set code that governs how records are stored in a repository also governs the display of repository containers and objects as object records in the user interface.  Though you likely wouldn’t choose to, it’s possible to use a different element set code for each repository and in turn display records from each repository differently to users.  A less extreme, more scaleable approach would identify repositories whose records share characteristics such as granularity or format and use the metadata code structure to display objects in those repositories similarly. A couple of examples will suggest possibilities along these lines.  The repositories we’ve created so far include one that contains records for digitized pamphlets from the May Anti-Slavery Collection.  Another repository contains records for the 800-plus reference resources in the e-Reference Collection that are currently available through the Cornell Library Gateway. 
This is an object record display for one of the pamphlets in the May Anti-Slavery Collection.  We’re planning to import the pamphlet records for this collection into a single repository, which, for a relatively homogeneous repository such as this one, opens up some opportunities with regard to display.  The rather plain display you see here is a carrier for the metadata and little more.  But because all of the contents of this repository are pamphlets, and as such are items of relatively small granularity within the overall scheme of the collections we are offering, we can use the display to convey those characteristics of format and granularity in words, icons, or both.
This is the object record for the Core Historical Literature of Agriculture, one of the resources in the Cornell Gateway e-Reference Collection.  Though the e-Reference records are also loaded in a single repository, the granularity of this repository’s contents varies more widely than that of the pamphlet collection.  And though this resource is represented as a single object within ENCompass, it is actually a collection unto itself.  Though the e-Reference Collection is relatively heterogeneous, resources in it do share some common characteristics that can be conveyed through the display.  As reference resources, they tend to bring bodies of information together in a single point of entry, and, as such, they tend to be of larger granularity, certainly larger than pamphlets.  We are developing a logo to represent the e-Reference Collection in our Library Gateway, so, at minimum, we could include that logo in the ENCompass object display for e-Reference resources to alert users to the kind of resource they are seeing on the screen.
While the repository structure in ENCompass controls how object records are stored and displayed in the system, the hierarchical collection structure determines how digital objects are presented to users in the searching and browsing interface.  By mapping metadata element sets, as they are stored in repositories, to unqualified Dublin Core and by arranging object records in hierarchies of collections and containers, we can offer users the option of searching across collections or performing structured browses within collections.  The hierarchy you see here is a simplification of the collection structure as it appears in the ENCompass staff interface.  Next, Sarah is going to discuss the collection structure from the perspective of the ENCompass Web client in order to suggest ways to adapt the client to better communicate the relationships among objects in the collection structure.
So how do we ensure that the search-and-browse interface presents content to our users in a meaningful way?  Part of our challenge is in portraying multidimensional data in the two-dimensional world of the computer screen.  Edward Tufte refers to this as “flatland,” and he emphasizes that our goal must be to escape the flatland and enrich “the density of display.”  He goes on, “by giving focus over to the data rather than data-containers, these design strategies are transparent and self-effacing in character…designs so good that they are invisible.” (p.12, 33)  The above two maps illustrate Tufte’s point:  the one on the left has flat elements - color and texture are equal, and the result is a blur.  The one on the right distinguishes separate layers of information by creating a rich language of color, light and shape.
Currently in the Encompass interface, we display what Tufte referred to as data containers.  If we look in the Browse mode, we can see collections on the first screen (shown here) and then click on them to reveal containers and objects.  In search mode, we can see the label for the type of data container in brackets (container, object, etc.) as we peruse our results list.
Given that we are still foregrounding data containers in the current interface, how well are we providing visual representations of them? Lets see how a results list from a search displays.

What we call “containers” in ENCompass are displaying as equals with objects.  We are using bracketed text alone to describe different data container types.  We are not providing graphic aids, such as, icons, font formatting and use of indentation or white space.  In Search mode, all discrete “things” look identical graphically, and the language we are using to describe them is likely opaque to our users.
In Browse mode, at least users have some “navigational aid” as they work their way deeper into the data containers.  If we wanted to browse the Audio Samples collection from the front screen you saw earlier and decided to look at the Ornithology subcategory (or “container), we would see this screen, which lists objects in the container.
If we want to go back to the “parent collection,” or Audio Samples - Ornithology, we can click on the “Back to Parents” link.
Notice the wording in the red bar: “Parents of Evening Grosbeak.”  This gives users some indication of where Evening Grosbeak belongs, but there is no sense of context, of how this object fits with the bigger picture.
So how can we break out of this “flatland?”  How can we make the data speak for itself?  How do we convey complex relationships which extend beyond two-dimensional space and indeed can mutate depending on user manipulation?  Further, if in future we offer a customizable interface, a sort of “My ENCompass,” one that will create a profile based on search history and habits, how do we help users make sense of relationships between “things” as they change? 
Jakob Nielsen goes so far as to say that what users most want is a clear means of navigation:  Where am I?  Where have I been?  Where can I go? And, how can I get there again? In fact, his research indicates  it is not all that clear that users need to understand the structure of their records in order to successfully execute searches.
We might agree that casual users are not so interested in hierarchical structure, i.e., what object fits in which container, collection, etc.  However, some more sophisticated users, such as researchers, do find hierarchy useful.  What’s more, even casual users show an interest in hierarchy when it can help them find more relevant information.  Information retrieval research suggests that users employ Search and Browse functions as means to learn the system better.  Users who favor searching want to be able to make sense of where they are once they finish a search, and Browse gives them some context.  Users who favor browsing may use the gained hierarchical knowledge to begin a search.  For users who combine Search and Browse, an interface which offers both is essential.
The ENCompass 1.1 interface integrates Search and Browse on the same screen.  Users can enter their search term above or browse below. This ability to search and browse within the same screen is powerful, but it can also be confusing. In addition, once users have entered search terms on this screen, they can browse from the next screen.  As they proceed through several screens, they may not understand the dual capability and may want to retrace their footsteps.  The clue I showed you in the red bar for “Parents of Evening Grosbeak” would only help users identify their very last step and would not lead them back over their entire navigational path.  Release 2.0 will include some navigational aids, such as breadcrumbs, particularly at the object level.  It will also include separate screens for Search and Browse, which should meet the needs of both browsers and searchers better. 
Whether or not we agree that different types of users care about hierarchy to varying degrees, we might postulate that all are interested in what type of “thing” they are viewing.  Whereas “traditional” library users were more concerned with whether they were viewing a book or an article, now users are dealing with differences like sound versus graphic versus text file.  They may need to know what application is required to open an object, or how large the object is.  In an environment like ENCompass, where users are searching heterogeneous collections, one way we can provide clarity is to add visual cues for formats of “things,” or objects. 
Currently in ENCompass,the only way to ascertain format is through metadata in the object record.  Depending on how the record is set up, the format field or the description field might have text indicating we are looking at a sound file or a text file, etc.  But there is no systematic, prominent mechanism in place to display format type.
Lets look at an example of an interface which uses icons to convey format types:  OCLC FirstSearch's WorldCat.  The icons represent metaphors which are fairly universal, but just in case, text also displays, and definitions of each format are available on another screen.  Note the legend near the top of the screen and the icons alongside the records.  Users can click on individual search results or on format icons to retrieve a list of objects with the same format. 
How can we learn from this in ENCompass?  Unlike OCLC's WorldCat, we are not dealing with MARC records alone nor “things” of the same granularity.  We are adding records in varying metadata element sets, as Marty mentioned earlier.  In MARC, we could conceivably pull format information from the Leader. Where would we do this for non-MARC records?  We would need to examine the feasibility of pulling metadata from certain fields of either the repository, container or object record, depending on the makeup of the collection.  More homogenous repositories would allow for this kind of treatment better.  As an initial step, we could at least display format for MARC records, thereby adding a sort of third dimension to the search results screen.
 
Displaying format may be one means of managing heterogeneous collections and thus breaking out of flatland in future, but in the meantime, we are focusing on other display techniques.  Marty showed you the relationship manager in the ENCompass staff client a few minutes ago.  You saw it uses an expandable structure and resembles applications like Windows Explorer.  We can click on recognizable icons like file folders to open and reveal logically arranged contents which in turn can be expanded further.
 
Why not bring this widely recognized expanding outline technique to our user interface?  In fact, the 2.0 version will employ a tiered folder system in the Browse mode.  We will likely have textual rather than iconic representations of folders, but the look and functionality will be the same.  Merely by indenting and arranging hypertext in an outline form, we can suggest the file folder structure, which lets the data entities and relationships speak for themselves, instead of hiding them behind flat displays and opaque phrases like “get children.”  As I mentioned earlier, 2.0 will also provide "breadcrumbs" at the top of each object record, mirroring the users' path through the hierarchical folder structure.
 
While the expanding folder works well as a browse tool for users, it poses a scalability problem in the long term.  Users will likely tire of clicking through level after level of hierarchy.  In addition, they will expect a manageable number of categories at the top level of display and will be less likely to scroll past the first visible screen.  How we choose our core list of categories and how we continue to fit new collections into our scheme will define users' success in browsing.
To conclude, the questions we’ve raised today point to three areas to explore as we continue our development effort:
We can investigate enriching object record displays to enable them to carry such information as the relative granularity of the entities presented. 
With the advent of XSL and hierarchical displays in the search-and-browse interface, we can experiment with different methods of presenting object relationships to our users.
And, finally, for use throughout the public interface, we can develop an intuitive vocabulary of terms and icons that represent digital objects and the relationships among them.
Thank you.
