Context
We have been adding online resources in increasing numbers for 10 years.
Currently 200+ online databases with thousands of ejournals, ebooks.
Use is increasing and expectations from users are high.
Information systems provides inadequate management functions
History
Original automated library systems were for managing cataloging records,  acquisitions, and purchasing transactions.
Our local systems were developed prior to client/server computing and the Web.
Most vendor systems developed prior to Internet/Web - not optimized for Web environment.
Old System...
History - continued
We began struggling with “full text” subs in 1991 with Lexis/Nexis.
We began dealing with Internet based information as ejournals emerged via Telnet, Gopher, WAIS in 1992.
By 1996, it became clear that our legacy systems were not capable of managing our rapidly growing digital collections of web-based products and locally created digital content (including metadata).
New Systems
Legacy Systems Environment
OPAC is offshoot of locally built central circ system - our own legacy systems
All systems designed for managing metadata
Our current website now a legacy system
Website is part of our dis-integrated library systems environment.
We have used the web as the modern user interface to our licensed and locally created information services.
We have attempted to meet our current management information systems needs through our website.
Needs
Current systems should be built for managing not just digital catalog and purchasing records - but digital content objects themselves.
Should be designed to manage and provide direct access to images, documents, audio, video.
Should be designed for use on the global Web.
Vendors are adapting to digital environment
But publishers must move, too.
Book publishers and library automation vendors are working together...

Preliminary Solution
Why FileMaker Pro?
Relatively cheap, university discount for client/server software.
Client available for PC and Apple desktop computers.
Easy to learn and manipulate.
FileMaker Database Uses
Originally meant to assist with creation of web pages.
Data exported and “munched” to create raw HTML.
Initial fields
Resource name
Vendor/publisher
URL
Description
Coverage notes
Other info for web pages
FileMaker Pro Database Evolution
Shared on network, multiple users.
Web accessible using Lasso.
(Blue World Communications, http://www.blueworld.com).
Running on dedicated NT server running FileMaker Server.
Evolving Uses
Subscription info.
Vendor/publisher contact info.
License tracking.
Allow users to search for and get status of licenses and view agreements.
Real-time database notifications to end-users re: availability and downtime
Slide 17
Limitations
Semi-relational.
Limited access, need client or specially designed web interface.
Problems building dynamic/static web pages directly from database.
Does not provide site management/evaluation tools.
Current Solution
LDAP (lightweight directory access protocol) database using iPlanet directory server running on Unix boxes.
Cheap
Allows directory management across multiple servers, server replication.
Optimized for reading of data.
Provides customizable site management and evaluation opportunities.
Multiple users with password access/different authority.
Could be made to be OAI and Dublin Core compliant.
