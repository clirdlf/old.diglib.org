Today I’d like to describe a project that we have been working on -off and on- at USC for the past year. 
Typical of most university systems, USC collections are organized in a myriad of formats and with inconsistent access rules.  The approach we have taken to providing access to this disparate set of collections is to develop a web services approach to content management and access.  The project, in a nutshell is a locally focused federated search and content management system.
USC researchers are producing content at an astronomical pace.  We have a high performance computing center, the Southern California Earthquake Center witch records data from thousands of ground motion sensors deployed throughout southern California to the school of cinematic arts producing more films per year than all of the Hollywood studios combined.  All of the systems on campus produce metadata of some sort.  We needed a way to bring that metadata together for search and access.  We do not have the ability within the library to collect all of this material, but we still have a role to play in providing access to content. 
We have a number of collaborators on campus who desire to work with digital archives, but they want to have control over the interface, search and access methodology.  Such as the Monastic Matrix, a collaborative repository of scholarly materials related to premodern Christian religious women. Which contains digitized texts, images, data, and new research on religious women in premodern Europe, all inter-linked and searchable texts, images, bibliography, and raw data simultaneously.  We currently provide the matrix with content management.  They do the rest. 
We have noticed a large increase in the use of library resources over the past few years, however the students and scholars are bypassing the catalog for other discovery tools.  The metadata in the ILS is the culmination of the collective intellectual life of the university.  That’s an incredibly rich resource that’s underused and like the collection it represents has value.  Scholars and patrons have increasing expectations for information delivery.  One of the goals of Project Gandhara is to make this resource more available to patrons where they seek information. Like everyone working in libraries these days, we also had a desire to add additional content into our ILS records and more from the current records.  Once we freed our records from the boundaries of the OPAC, we are able to rethink our bibliographic services.  We are able with this system to do things like bring in book art and editorial reviews from Amazon. 
Gandhara refers to an ancient region in northern Pakistan.  It was located on the spice road known for its fusion of cultures and art forms.  We felt that this crossroads of civilizations represents the nature of the project.
A very brief slide about the platform we have created. All of the component parts are opensource software.    The platform we have built is written in the struts framework. Struts is an open source framework for building web applications.  The core of which is a flexible control layer based on standard technologies like Java Servlets, JavaBeans, ResourceBundles, and Extensible Markup Language (XML).
We strive to take that flexibility to the architecture as well.  Our goal is to facilitate plugging in modules such as custom indexers.  Providing users the ability to create custom interfaces to Gandhadra.  Also we worked hard to maintain flexibiltiy of the platform itself for replacing components such as the full text engine. 
We are currently in phase one of the project.  Focused on proof of concept and library controlled systems.  I’ll describe a few important aspects of the project in detail and then get into a discussion of the architecture of the system.  Some content we directly ingest some content we harvest.  For example the catalog data we harvest.  It’s a 2 stage process taking the MARC records we transform them into MARC XML and then again into dublin core OAI xml.  At that point we ingest the descriptive metadata elements for the purpose of search.  Some content we simply ingest directly into the XML to store and search. 
Once we have the content we have a rather large set of content from various sources.  Our current system contains about 300k catalog records, 1 million web pages, about 150k records from the digital archive and about 600 records from 24/7 chat as well as content harvested from places like Amazon. We need a way to organize that content a bit.  We use the Internet Archive's open-source web crawler Heritrix to create compressed archives of the data.  Once we have the dataset from the crawl we use lucine, a text search engine library from Apache to create a full text index.  This index can also be crawled by google and other commercial search engines.   We also have a OAI-PMH compliant set of  Marc records.  One of the added benefits of the system is that we are able to migrate records from Gandhara into the ILS.  For instance records for our electronic thesis and dissertation collection within DSpace can be synchronized with the catalog.

This is the diagram of the system we currently run.  On the left of the diagram you see our current list of sources.  This list contains two integrated library systems, our digital archive, a crawl of the USC web and content from OCLC's QuestionPoint service.  The next column is a list of the current harvesters we run, the top two are harvesting data from two Integrated Library systems, we will be adding a third this fall, our web crawl and our OAI and knowledge base harvesters.  The two blue boxes in the middle is the real good bit.  The local cache and the platform.  I’ve described the platform already and I’ll get into the local cache in greater detail on the next slide, however the local cache is where we apply our custom XML wrapper.  The next column represents our platform where we use pluggable indexers and parsers and lucine libraries to create the full text index. For instance we index elements of our MARC records along with the full text as well as content like editorial reviews.  Once that is completed we then have a full text index for searching.   
So now let’s discuss the local cache and the XML wrapper.  It’s a simple schema which we use to wrap all of our content prior to indexing.  That content as you have seen comes in may formats but text, xml and html for the most part.  Within that local cache we have some control over display by using style sheets, rendering rules as well as store schema for XML in the cache.  In case we need some information about DTDs for style sheets and rendering rules. 
This is our XML dtd.  Some of the highlights are the description element which is pulled directly from the record, if there is no description from the page, for instance if it came from a web page, we use a summry or a text snippet.  The archive element indicates the location of the archive on the disk, as I mentioned earlier we are using heretrix to create compressed archives of our content, this element indicates where the archive is located. The offset element indicates the location within the archive of a particular piece of data.  The sourceurl is the link ot the original location of the record, for instance the catalog record.  The electronicurl will link directly to the content.  The content element is there in case some data we gather is in plain text.  We can simply place it in the wrapper.
