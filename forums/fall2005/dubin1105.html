Digital Preservation

  • This part of the project is about what the OAISRM calls "Representation Information"
  • "The information that maps a Data Object into more meaningful concepts"
  • The aim is to build models that aid in making inferences about digital resources.
  • METS is an XML application for encoding metadata about objects in digital libraries. METS files represent (diverse) efforts to express some levels of representation information in a machine-readable form.
  • We're interested in what we can learn from efforts to draw inferences about digital objects by reasoning with their METS encodings.

Markup Interpretation

  • XML applications overload a small number of generic syntactic relationships (e.g., parent/child) to represent a variety of specific semantic relationships.
  • Human beings correctly infer the meaning of markup, and these understandings inform the logic and design of applications that import, export, and transform XML.
  • But relying on tacit human understanding becomes problematic as collections age, become larger, and more heterogeneous.

Data Streams

METS markup has several ways to license the inference that a particular data stream exists.

<METS:file ID="DS1.0" CREATED="2002-05-20T06:32:00" MIMETYPE="image/x-mrsid-image" 
ADMID="TECH1 DIGIPROV1 SOURCE1 RIGHTS1" OWNERID="E" STATUS="A">
  <METS:FLocat LOCTYPE="URL" 
  xlink:href="http://iris.lib.virginia.edu/mrsid/mrsid_images/iva/archerd01.sid" 
  xlink:title="Exhibit Intro: Architectural Drawings, Pavilion III, University of Virginia"/>
</METS:file>

Data Streams

Data streams can be pointed to or encoded within the METS document.

<dmdSec ID="dmd003">
   <mdWrap MIMETYPE="application/marc" MDTYPE="MARC" LABEL="OPAC Record">
     <binData>MDI0ODdjam0gIDIyMDA1ODkgYSA0NU0wMDAxMDA...(etc.)
     </binData>
  </mdWrap>
</dmdSec>

Data Streams

Even a subtree of the METS document can be understood as a data stream.

<METS:dmdSecFedora ID="DC" STATUS="A">
 <METS:descMD ID="DC1.0" CREATED="2002-05-20T06:32:00">
  <METS:mdWrap MIMETYPE="text/xml" MDTYPE="OTHER" LABEL="DC Record for Exhibit Intro: 
  Pavillion III Architectural image object">
   <METS:xmlData>
     <oai_dc:dc xmlns:oai_dc="http://www.openarchives.org/OAI/2.0/oai_dc/" 
     xmlns:dc="http://purl.org/dc/elements/1.1/">
        <dc:title>Exhibit Intro: Architectural drawings, Pavilion III, 
         University of Virginia</dc:title>

Data Streams

Some interesting meronymic issues are raised by this example. Also problems of reasoning about files in file systems.

 <mets:fileGrp ID="FILEGROUP_PRELUDE1.2">
   <mets:file MIMETYPE="text/xml" ID="FILE1.2">
      <mets:FLocat LOCTYPE="URL"
            xlink:href="file:doctxt.xml#xpointer(/Section[1]/Section[2]/text())"/>
   </mets:file>
 </mets:fileGrp>

Parts or portions of Data Streams

The area element can be of type "time" or "polygon" but is supposed to represent portions of a file.

<fptr FILEID="FILE002">
  <area FILEID="FILE002" BEGIN="00:01:48"END="00:06:17"
  BETYPE="TIME" />
</fptr>

How we're doing this

  • The BECHAMEL System is written in Prolog, and is designed to support inferences about markup.
  • We create declarations about the kinds of things that exist, their properties, and the relationships that can hold among them.
  • We experiment with mapping rules of various kinds for instantiating objects, assigning property values, and establishing relationships.
  • The first steps involve building and experimenting with different models.
  • The knowledge structures we're creating are of the kind that can be serialized as RDF, OWL, etc.
  • Later applications that deploy this knowledge need not be written in Prolog.
